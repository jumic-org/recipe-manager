# Password change feature via Cognito SDK

This change adds a password change flow for authenticated users. A new `changePassword` method in `AuthService` wraps the Cognito SDK's `CognitoUser.changePassword`, a new standalone `ChangePasswordComponent` presents the form, a guarded route is registered at `/change-password`, a nav link surfaces it to authenticated users, and i18n strings are added in English and German.

**Watch for:** The component uses `OnPush` change detection but mutates local state (`loading`, `errorMessage`, `successMessage`) without triggering change detection — the UI will not update after form submission (confirmed). The form does not clear sensitive field values on success or error (confirmed). No test coverage exists (confirmed — no test runner is configured in this project).

## High-level view

The component has a change detection incompatibility: it uses `ChangeDetectionStrategy.OnPush` while mutating plain instance fields inside async subscription callbacks, which means Angular will not re-render the template after the password change completes or fails. The existing auth components have the same pattern, but the bug is real regardless of precedent.

On success, the form retains entered passwords (`currentPassword`, `newPassword`, `confirmNewPassword`) in DOM input fields. These are sensitive credentials that remain accessible until the user navigates away.

## OnPush vs mutable state in ChangePasswordComponent

The component declares `changeDetection: ChangeDetectionStrategy.OnPush`. The `onSubmit` method modifies `this.loading`, `this.errorMessage`, and `this.successMessage` inside the `subscribe` callbacks of an Observable that isn't bound through an `async` pipe. After the Cognito call resolves, Angular has no reason to re-render the component — the user will still see the "Changing..." button state and will never see the success or error message.

The existing `LoginComponent` and `RegisterComponent` have the same pattern (OnPush + mutable fields inside subscribe), so this is replicated from existing code. The fix is to inject `ChangeDetectorRef` and call `this.cdr.markForCheck()` after each state mutation, or refactor the state to signals.

```typescript
// In the subscribe handlers:
next: () => {
  this.loading = false;
  this.successMessage = this.translateService.instant('AUTH.CHANGE_PASSWORD.SUCCESS');
  this.cdr.markForCheck(); // <-- required for OnPush
},
error: (err) => {
  this.loading = false;
  this.errorMessage = err.message || 'Password change failed';
  this.cdr.markForCheck(); // <-- required for OnPush
}
```

## Form does not reset after success

On a successful password change, the form fields retain their values. These are sensitive credentials sitting in the DOM until the user navigates away. A `this.form.reset()` call after setting `successMessage` would clear them immediately and also disable the submit button (since the form becomes invalid), preventing accidental double-submission.
