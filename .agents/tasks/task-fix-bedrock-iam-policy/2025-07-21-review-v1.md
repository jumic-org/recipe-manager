# Fix Bedrock IAM policy ARN and Lambda timeout for recipe import

The recipe import feature calls Bedrock's `InvokeModel` with model ID `eu.amazon.nova-lite-v1:0`, which is a cross-region inference profile (indicated by the `eu.` prefix). The IAM policy was granting access to a `foundation-model` resource type, which doesn't match — causing an authorization failure surfaced as "Failed to extract recipe using AI model". The fix changes the ARN to the correct `inference-profile` resource type and bumps the Lambda timeout from 30s to 60s to accommodate the URL fetch plus inference latency.

**Watch for:** The IAM action is still `bedrock:InvokeModel` but inference profiles may require `bedrock:InvokeModel` on both the profile and underlying model depending on account configuration (possible). The wildcard in the account field (`*:*`) is intentionally broad.

## High-level view

The ARN fix corrects the resource type from `foundation-model` to `inference-profile` and populates the account field with a wildcard. This directly addresses the access-denied error path that produces the "Failed to extract recipe using AI model" message in the handler's catch block (line ~330 of handler.ts). The handler calls `InvokeModelCommand` with `modelId: 'eu.amazon.nova-lite-v1:0'` — the `eu.` prefix routes to a cross-region inference profile, not a direct foundation model endpoint, so the resource ARN in the policy must reflect that.

The timeout bump from 30s to 60s gives headroom for the sequential operations in `importRecipe`: an 8-second URL fetch timeout, followed by Bedrock inference that can take 10-30s depending on prompt size and model load. At 30s total, the function was at risk of timing out on valid requests even after the IAM fix.

## Inference profile ARN structure

The original ARN `arn:aws:bedrock:*::foundation-model/eu.amazon.nova-lite-v1:0` has two problems for an inference profile resource:

1. The resource type `foundation-model` doesn't match what the service evaluates when the call routes through a cross-region inference profile. The correct resource type is `inference-profile`.
2. The account field was empty (`::`) — foundation models are service-global so they omit the account, but inference profiles are account-scoped resources requiring an account ID or wildcard.

The corrected ARN `arn:aws:bedrock:*:*:inference-profile/eu.amazon.nova-lite-v1:0` addresses both. The double wildcard (`*:*` for region and account) is appropriate here because cross-region profiles route requests to multiple regions and the caller shouldn't need to enumerate every possible target region/account combination.

One gap worth noting: some Bedrock configurations require permission on both the inference profile ARN *and* the underlying foundation model ARN when invoking through a profile. If this deployment encounters an access-denied error again after this fix, adding a second resource entry for `arn:aws:bedrock:*::foundation-model/amazon.nova-lite-v1:0` (without the `eu.` prefix) would resolve it. This is configuration-dependent and may not apply here, but it's the most common follow-up issue with inference profile permissions.

## Timeout budget

The `importRecipe` function performs two sequential I/O operations:

```
URL fetch (8s AbortSignal timeout) -> Bedrock InvokeModel (no explicit timeout)
```

At 30s Lambda timeout, after an 8s fetch, only 22s remained for Bedrock inference. Nova Lite typically responds in 5-15s, but under load or with large prompts (the prompt includes up to 10KB of page content plus ~3KB of examples), response times can approach 20-25s. The bump to 60s provides reasonable headroom without being wasteful.

The Bedrock call itself has no client-side timeout configured — it relies entirely on the Lambda execution timeout for cancellation. If the Lambda times out mid-inference, the Bedrock request continues server-side but the response is discarded. This isn't a correctness issue but is worth knowing for cost attribution: a timed-out Lambda still incurs the full Bedrock inference cost.

<summary>Issues (2)</summary>

1. **Inference profile may also need foundation model permission** — If the account's Bedrock configuration requires dual-resource authorization for inference profiles, the policy will still deny access. Adding `arn:aws:bedrock:*::foundation-model/amazon.nova-lite-v1:0` as a second resource would cover this case. (possible)
2. **No client-side timeout on Bedrock call** — The `InvokeModelCommand` has no explicit timeout; only the Lambda execution deadline bounds it. A hung Bedrock call will consume the full 60s before failing. Consider adding an `AbortSignal.timeout()` to the Bedrock invocation similar to the URL fetch. (likely)
