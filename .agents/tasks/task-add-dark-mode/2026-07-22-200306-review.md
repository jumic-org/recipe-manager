# Dark mode via CSS custom properties and ThemeService (v2)

A dark mode toggle was added to the Angular frontend. A root-level `ThemeService` manages a `'light' | 'dark'` state string, persists it to localStorage, detects OS preference via `prefers-color-scheme` on first visit, and applies a `dark` class to `document.documentElement`. CSS custom properties defined in `:root` and `:root.dark` blocks in `styles.scss` handle color switching. All eight component style files were migrated from hardcoded hex values to these custom properties, including a `--rm-danger-surface` token for the remove button. The toggle button lives in the app header with proper aria-labels and i18n support.

**Watch for:** Plain getter with OnPush change detection won't re-render if theme is changed externally — not a bug today but prevents future cross-tab sync without rework (possible). Transition declarations on body and `.app-shell` could produce a flash if bootstrap timing changes under SSR (possible).

**Verdict**: APPROVED

## High-level view

Both confirmed issues from v1 are resolved: `.btn-remove` uses `var(--rm-danger-surface)` backed by a new variable in both `:root` and `:root.dark`, and the ThemeService falls through to `window.matchMedia('(prefers-color-scheme: dark)')` when localStorage has no stored value.

The two remaining observations are forward-looking and non-blocking. The getter-over-private-field pattern in an OnPush component won't scale if the service ever needs to react to external events (cross-tab sync, runtime OS preference changes). The transition-on-load concern is theoretical given the SPA bootstrap sequence but would surface under SSR.

1. **OnPush reactivity gap** — `themeService.currentTheme` is a plain getter in an OnPush component. External state changes (cross-tab `storage` events, OS preference changes at runtime) won't trigger re-render. Converting to an Angular signal would future-proof this. (possible)
2. **Transition flash under SSR** — `transition: background-color 0.3s, color 0.3s` on `body` and `.app-shell` fires on initial paint if the `dark` class is applied after first render. Currently safe because the service constructor runs before template rendering, but SSR or deferred initialization would expose a light-to-dark flicker. (possible)

## OnPush reactivity and signals

If the service later adds a `storage` event listener (cross-tab sync) or a `matchMedia` change listener (to follow OS preference in real time), the template in `app.component.html` would go stale — there's no signal, observable, or `markForCheck()` to notify the OnPush component. Angular 21 signals are the idiomatic path:

```typescript
readonly currentTheme = signal<Theme>(this.resolveInitialTheme());
```

No external mutation path exists today, so this is a design note for future extension rather than a current defect.

## Transition timing under SSR

The `transition: background-color 0.3s, color 0.3s` declarations on `body` and `.app-shell` smooth the toggle animation but also mean the transition fires on initial load if there's any gap between first paint and the `dark` class appearing on `<html>`. In the current SPA setup the gap is effectively zero (constructor runs before any template paints). Under SSR the server-rendered HTML won't have the `dark` class — the client-side service would add it post-hydration, producing a visible 300ms fade. Mitigation: scope the transition behind a `theme-ready` class added only after the first user-initiated toggle.
