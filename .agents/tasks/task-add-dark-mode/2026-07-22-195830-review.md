# Dark mode via CSS custom properties and ThemeService

A dark mode toggle was added to the Angular frontend. A root-level `ThemeService` manages a `'light' | 'dark'` state string, persists it to localStorage, and applies a `dark` class to `document.documentElement`. CSS custom properties defined in `:root` and `:root.dark` blocks in `styles.scss` handle the actual color switching. All eight component style files were migrated from hardcoded hex values to these custom properties. The toggle button lives in the app header alongside the existing language switcher, with proper aria-labels and i18n support.

**Watch for:** Incomplete color migration leaves a hardcoded `#fdeaea` that won't respond to dark mode (confirmed). No `prefers-color-scheme` support means first-time visitors always see light mode regardless of OS preference (confirmed). Plain getter with OnPush change detection won't support future external theme changes without rework (possible).

## High-level view

The theming architecture is a class-on-html-root approach — ThemeService owns the toggle state and DOM class manipulation; the CSS layer does all the visual work through variable overrides. No per-component logic for color switching, just a single class toggle at runtime.

The 16-token variable palette spans backgrounds, text, borders, shadows, inputs, and semantic colors. The migration across components is nearly complete, with one leftover hardcoded background value in `recipe-form` that will appear as a jarring light-mode artifact when dark mode is active.

The service doesn't consult `prefers-color-scheme`, so users whose OS is set to dark mode will still land on light mode on first visit — a single `matchMedia` check as the localStorage fallback would fix this.

## Incomplete color migration in recipe-form

The diff touched `.btn-remove` to replace `color: #c0392b` with `var(--rm-danger)` but left `background: #fdeaea` as a hardcoded hex value on the same rule. In dark mode, this light pink background will stand out against the dark surface. The same pattern was properly converted elsewhere (e.g., `.btn-add` uses `var(--rm-primary-surface)`), so the omission is likely accidental. A `--rm-danger-surface` variable (light: `#fdeaea`, dark: something like `#3d2626`) would follow the existing pattern.

## No `prefers-color-scheme` fallback

ThemeService defaults to `'light'` when localStorage has no stored value:

```typescript
const stored = localStorage.getItem(ThemeService.STORAGE_KEY);
this.theme = stored === 'dark' ? 'dark' : 'light';
```

A more complete default would check the user's OS preference:

```typescript
const stored = localStorage.getItem(ThemeService.STORAGE_KEY);
if (stored === 'dark' || stored === 'light') {
  this.theme = stored;
} else {
  this.theme = window.matchMedia('(prefers-color-scheme: dark)').matches ? 'dark' : 'light';
}
```

## Reactivity gap with OnPush

The template reads `themeService.currentTheme` — a getter returning a private field — inside a component with `ChangeDetectionStrategy.OnPush`. If this service later needs to react to external events (a `storage` event from another tab, or a `matchMedia` listener for OS preference changes), the template won't re-render because there's no signal, observable, or `markForCheck()` call. Angular 21 has signals available; converting `currentTheme` to a `signal<Theme>()` would make the service reactive-ready and is idiomatic for Angular's current direction.

## Transition timing on initial load

The `transition: background-color 0.3s, color 0.3s` declarations on `body` and `.app-shell` provide a smooth toggle animation but also mean the transition fires on initial load if there's any gap between first paint and the `dark` class being applied. Because ThemeService is instantiated during Angular bootstrap (before the first template renders), the class should already be on `<html>` before the component tree paints. But if the app ever moves to SSR or the service initialization is deferred, this could produce a visible light-to-dark flash. A CSS-only mitigation: scope the transition to only apply after a user interaction (e.g., add a `theme-ready` class after the first toggle).
