# i18n via ngx-translate with EN/DE language switcher

This change wires up `@ngx-translate/core` v18 and `@ngx-translate/http-loader` v18 to externalize all user-visible strings into JSON translation files. English and German translations live at `assets/i18n/{en,de}.json` with a hierarchical key structure (APP, AUTH, RECIPES). Every component template switches from hardcoded English to `TranslatePipe` lookups, and a language switcher in the app header toggles between EN and DE at runtime.

Watch for: German translations use ASCII umlaut approximations ("ue"/"oe"/"ae") instead of actual Unicode characters, producing visibly incorrect German (confirmed); several error-path strings remain hardcoded English and will not translate (confirmed); language selection is ephemeral and resets on reload (confirmed); `instant()` in OnPush components assigns to plain properties without triggering change detection (likely).

## High-level view

Three error-handler callbacks and one fallback string in `recipe-list` still use hardcoded English (`'Failed to load recipes'`, etc.), making them invisible to the translation system. These are the same pattern correctly translated elsewhere (e.g., `RECIPES.DETAIL.NOT_FOUND`), so the omission appears accidental.

The German translation file uses ASCII digraphs for every umlaut character — "Passwoerter" instead of "Passwörter", "Zurueck" instead of "Zurück", "Loeschen" instead of "Löschen". This convention belongs to legacy systems that cannot handle Unicode, not a UTF-8 web application. Every German string containing ä, ö, ü, or ß is affected.

The language preference lives only in `AppComponent.currentLang` and is not persisted. Every page reload or new tab resets to English regardless of the user's prior choice.

Recipe components use `ChangeDetectionStrategy.OnPush` and assign `this.error`/`this.loading` inside `subscribe()` callbacks without calling `markForCheck()`. This is a pre-existing pattern, but the PR extends it with new `translateService.instant(...)` assignments that may not render.

## Untranslated error paths

Error messages assigned in TypeScript error callbacks remain in English:

```typescript
// recipe-list.component.ts
error: (err) => {
  this.error = err.message || 'Failed to load recipes';
};

// recipe-detail.component.ts
error: (err) => {
  this.error = err.message || 'Failed to load recipe';
};
// and:
error: (err) => {
  this.error = err.message || 'Failed to delete recipe';
};

// recipe-form.component.ts
error: (err) => {
  this.errorMessage = err.message || 'Failed to load recipe';
};
```

The `confirm.component.ts` and `recipe-detail.component.ts` (for the "not found" case) already demonstrate the correct pattern using `this.translateService.instant('KEY')`. These four remaining fallback strings need the same treatment. `err.message` itself comes from the server/HTTP layer and would also display in English—translating server errors is a larger concern outside this PR's scope.

## German translation quality

Every umlaut in `de.json` is replaced with a two-character ASCII digraph:

- "Passwörter" → "Passwoerter"
- "Zurück" → "Zurueck"
- "Bestätigung" → "Bestaetigung"
- "Löschen" → "Loeschen"
- "möchten" → "moechten"
- "Hinzufügen" → "Hinzufuegen"

This is the transliteration convention for systems that cannot handle Unicode. The JSON files are UTF-8; the proper characters should be used directly. This affects nearly every string in `de.json` that contains a German-specific character.

## Language persistence gap

`AppComponent` holds `currentLang` as a plain property initialized to `'en'` and calls `this.translate.use('en')` in the constructor. There is no attempt to read from `localStorage`, `sessionStorage`, or `navigator.language`. A user who selects German will be reset to English on every full page reload or new tab.

Minimal fix: persist to `localStorage` on switch, read on construction, fall back to `'en'` if no stored preference exists.

## OnPush change detection with imperative state

All recipe components use `ChangeDetectionStrategy.OnPush` and assign `this.loading`, `this.error`, etc. inside `subscribe()` callbacks. Angular's OnPush strategy only re-renders when an `@Input` reference changes, an event originates from the component/children, or an async pipe fires. None of these conditions are met inside an HTTP `subscribe` callback.

This is a pre-existing architectural issue, but the PR adds new `this.error = this.translateService.instant(...)` assignments in the same pattern. The translated error string may not render without an explicit `markForCheck()` call.

## `instant()` timing consideration

`TranslateService.instant()` is synchronous and returns the raw key if translations haven't been loaded yet. The `provideTranslateHttpLoader` fires an HTTP request at application startup. Because `APP_INITIALIZER` (for config loading) blocks bootstrap but translation loading happens in parallel—not sequenced after it—there's a small race window where `instant()` could return the key literal instead of the translated string. In practice this window is narrow (users must navigate before error handlers fire), but it's worth noting.
