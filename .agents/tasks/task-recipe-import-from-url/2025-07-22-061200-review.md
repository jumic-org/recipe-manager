# Recipe import from URL via Bedrock extraction (v2)

All five issues from the v1 review have been addressed: SSRF protection via HTTPS-only + private IP blocking, fetch timeout with `AbortSignal.timeout(8000)`, Bedrock output validation through `validateRecipeInput`, prompt injection mitigation with `<PAGE_CONTENT>` delimiters, and client-side HTTPS pattern validation. The remaining concerns are non-blocking for a personal recipe manager behind Cognito auth.

**Watch for:** DNS rebinding bypasses the IP blocklist since `isPrivateOrReservedHost` checks the hostname string but not the resolved IP address (likely). No timeout on the Bedrock call means it could consume the remaining ~22s of Lambda budget with no way to surface a meaningful error (confirmed). The `validateRecipeInput` function is shallow — it checks four fields exist but doesn't constrain string lengths or array sizes (confirmed).

**Verdict**: APPROVED

## High-level view

The SSRF mitigation covers literal IP addresses, `localhost` variants, and IPv6 loopback/link-local, but does not resolve the domain to verify the actual target IP. A DNS rebinding attack remains theoretically possible. Low practical risk given Cognito auth gate and IMDS v2 on modern Lambda runtimes.

The Bedrock output validation reuses `validateRecipeInput`, which asserts `title` is a non-empty string, `servings` is a number, and `ingredients`/`instructions` are arrays. It doesn't enforce length limits or array bounds, but the 4096 `maxTokens` cap and 10,000-character page content limit naturally bound output size to ~16KB.

The Bedrock call has no per-operation timeout. The fetch gets 8s, but Bedrock inference under throttling could exceed the remaining budget. The Lambda's 30s ceiling is the only backstop, producing a generic API Gateway 503 rather than a descriptive 502.

## SSRF protection: residual DNS rebinding gap

The `isPrivateOrReservedHost` function covers RFC 1918 ranges, loopback, link-local (including AWS IMDS at 169.254.x.x), and `0.0.0.0`. The residual gap: an attacker registers `evil.example.com` resolving to `169.254.169.254`. The `new URL(url)` parse extracts `evil.example.com` as the hostname, which passes the string-based check. The subsequent `fetch()` resolves the domain and connects to the metadata IP.

IMDS v2 requires a PUT to obtain a session token before any GET returns data, and the Lambda's fetch follows the Bedrock prompt path (it reads the body as text and feeds it to the model), so credential exfiltration requires the model to echo back the metadata content in its JSON response — an unlikely but not impossible scenario via prompt injection. If this moves to a multi-tenant model, post-resolution IP validation becomes necessary.

## Bedrock timeout gap

The Bedrock `InvokeModelCommand` has no timeout:

```typescript
const bedrockResponse = await bedrockClient.send(
  new InvokeModelCommand({ ... }),
);
```

If Bedrock throttles or the model takes long to generate 4096 tokens, this call could run 20+ seconds. The Lambda's 30s ceiling fires, API Gateway returns a 503 with no body. Adding `{ abortSignal: AbortSignal.timeout(15000) }` to `.send()` would let the handler catch the timeout and return a descriptive 502.

## Validation depth on Bedrock output

`validateRecipeInput` catches structurally broken responses (missing title, non-numeric servings, ingredients not an array). It does not check string lengths, array element shapes, or numeric ranges (negative servings, impossibly large prep times). The 4096 `maxTokens` cap bounds absolute output to ~16KB of JSON, so this is a quality concern rather than a storage bomb risk.
