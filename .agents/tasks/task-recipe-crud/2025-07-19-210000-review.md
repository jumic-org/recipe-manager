# Full-stack serverless recipe CRUD with Cognito auth

A complete recipe management application built as an Nx monorepo: shared data model library, DynamoDB-backed Lambda CRUD handler, CDK infrastructure stack (DynamoDB, Lambda, API Gateway with Cognito authorizer, S3+CloudFront frontend hosting, KMS encryption), and an Angular 21 SPA with Cognito authentication flows. The approach is single-Lambda routing with single-table DynamoDB, tenant-isolated by Cognito `sub` claim as partition key.

**Watch for:** API response shape mismatch between Lambda and Angular service (confirmed), no input validation on the Lambda beyond body-exists checks (confirmed), `deleteRecipe` fails silently on condition-check errors instead of returning 404 (confirmed), CORS allows all origins in production (confirmed), auth guard relies on synchronous in-memory state that resets on page refresh race conditions (likely).

## High-level view

The Lambda handler routes all CRUD operations internally via path and method matching. Tenant isolation is enforced at the DynamoDB key level by extracting `userId` from the Cognito authorizer claims. The handler trusts the request body shape entirely — no runtime validation — which means malformed payloads will write garbage to DynamoDB or throw at attribute-access time.

The Angular frontend talks to the API through an `HttpClient` service, but the service expects flat `Recipe[]` and `Recipe` responses while the Lambda wraps them in `{ recipes: [...] }` and `{ recipe: {...} }` envelopes. This will cause runtime failures when the frontend tries to render API responses.

The auth implementation uses `amazon-cognito-identity-js` directly with a `BehaviorSubject`-based state. The auth guard checks synchronous state, which correctly reflects session validity after the constructor's async check completes — but there's a window on cold page loads where the guard may redirect before the session check resolves.

The CDK stack uses KMS encryption on both DynamoDB and S3, OAC for CloudFront, and Cognito authorizer on all API methods — but CORS is configured with wildcard origins on both the Lambda response headers and API Gateway, which should be restricted before production use.

<details>
<summary>Issues (7)</summary>

1. **API response envelope mismatch** — Lambda returns `{ recipes: [...] }` and `{ recipe: {...} }` but `RecipeService` expects unwrapped `Recipe[]` and `Recipe`. Extract the nested property in the service, or remove the envelope from the Lambda.
2. **No request body validation** — `createRecipe` and `updateRecipe` cast `JSON.parse(event.body)` directly to the input type with no runtime checks. Validate required fields or use a schema library to reject malformed payloads before writing to DynamoDB.
3. **Delete swallows ConditionalCheckFailedException** — `deleteRecipe` doesn't catch the DynamoDB condition-check error, so a delete of a non-existent recipe returns a 500 instead of 404.
4. **CORS wildcard in production** — `Access-Control-Allow-Origin: *` on the Lambda and `Cors.ALL_ORIGINS` on API Gateway. Restrict to the CloudFront domain for production deployments.
5. **Auth guard race on cold load** — `authGuard` calls `isAuthenticatedSync()` which returns `false` until the async `getSession` callback fires. A direct navigation to a protected route on page load may redirect to login even with a valid session.
6. **OnPush change detection without signals or async pipe** — `RecipeListComponent` and `RecipeDetailComponent` mutate component fields inside subscribe callbacks. With `OnPush`, the view won't re-render unless something triggers change detection (the zone will usually cover it, but it's fragile for future zoneless migration).
7. **No test coverage** — The entire change ships without a single test file. The steering docs describe a testing strategy but no tests exist.

## API response shape vs. frontend expectations

The Lambda wraps all responses in an object envelope:

```typescript
// handler.ts — listRecipes
return response(200, { recipes });  // { recipes: Recipe[] }

// handler.ts — getRecipe
return response(200, { recipe: result.Item as Recipe });  // { recipe: Recipe }

// handler.ts — createRecipe
return response(201, { recipe });  // { recipe: Recipe }
```

The Angular `RecipeService` declares return types as if the response IS the entity:

```typescript
getRecipes(): Observable<Recipe[]> {
  return this.http.get<Recipe[]>(this.baseUrl);
}
getRecipe(id: string): Observable<Recipe> {
  return this.http.get<Recipe>(`${this.baseUrl}/${id}`);
}
```

At runtime, `getRecipes()` will return an object `{ recipes: [...] }` cast as `Recipe[]`. The list component assigns this to `this.recipes` and iterates it in the template, which will fail — an object is not iterable. Similarly, `getRecipe()` returns `{ recipe: {...} }` but the component expects a `Recipe` directly. This is a functional bug that will break all read operations.

## Input validation gap in the Lambda

`createRecipe` and `updateRecipe` parse the body and immediately trust it:

```typescript
const input: CreateRecipeInput = JSON.parse(event.body);
```

TypeScript type annotations are erased at runtime. If a client sends `{}`, the handler will write an item to DynamoDB with `undefined` for every field (title, ingredients, etc.) or crash on the update expression with unhelpful DynamoDB errors. There's no validation that required fields like `title`, `servings`, or `ingredients` are present, no check that arrays are actually arrays, and no sanitization of string lengths.

For a Cognito-protected API this is lower-severity (only authenticated users can reach it), but it still allows a single malformed request to corrupt data that the frontend will then fail to render.

## Delete error propagation

The `deleteRecipe` function uses a `ConditionExpression` to ensure the item exists:

```typescript
await docClient.send(
  new DeleteCommand({
    Key: { userId, id: recipeId },
    ConditionExpression: 'attribute_exists(userId) AND attribute_exists(id)'
  })
);
return response(204);
```

When the item doesn't exist, DynamoDB throws `ConditionalCheckFailedException`. This exception isn't caught within `deleteRecipe`, so it bubbles to the top-level catch and returns a generic 500. The `updateRecipe` function has the same condition expression — if the condition fails, the `UpdateCommand` also throws rather than returning null attributes. Both operations need explicit handling of `ConditionalCheckFailedException` to return 404.

## CORS posture

The Lambda hardcodes `Access-Control-Allow-Origin: *` and the CDK stack configures `Cors.ALL_ORIGINS` on the RestApi. For an authenticated API, any origin can make requests (the Cognito token is the real protection, not CORS). Since the interceptor doesn't set `withCredentials`, this won't break today, but wildcard CORS on an authenticated API should be tightened to the CloudFront domain before production use.

## Auth guard timing

```typescript
// auth.service.ts constructor
const currentUser = this.userPool.getCurrentUser();
if (currentUser) {
  currentUser.getSession((err, session) => {
    if (!err && session && session.isValid()) {
      this.authenticatedSubject.next(true);
    }
  });
}
```

```typescript
// auth.guard.ts
export const authGuard: CanActivateFn = () => {
  if (authService.isAuthenticatedSync()) { return true; }
  return router.createUrlTree(['/login']);
};
```

`getSession` is async (it may hit network to refresh an expired token). The guard reads `isAuthenticatedSync()` synchronously. On a cold page load with a direct URL to `/recipes`, Angular instantiates `AuthService`, starts the async session check, and immediately evaluates the guard — which sees `false` because the callback hasn't fired yet. The user gets bounced to login, then the session resolves. The fix is to make the guard return an `Observable<boolean>` that waits for the session check to complete.

## Change detection and state mutation

All recipe components use `OnPush` change detection but mutate local fields (`this.recipes`, `this.loading`, `this.error`) inside `subscribe` callbacks without explicitly marking for check. This works today because Zone.js patches `subscribe` and triggers change detection, but breaks under zoneless Angular. Using the `async` pipe or `signal()`-based state would make this robust.

## Missing test coverage

No test files ship with this change. The Lambda handler's routing logic, the DynamoDB command construction, and the auth interceptor token-attachment logic are all straightforward to unit-test and would catch the envelope mismatch immediately.
