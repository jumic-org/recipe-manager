# Full-stack serverless recipe CRUD with Cognito auth (v2)

The four critical issues from v1 have been addressed: `RecipeService` now correctly extracts from response envelopes via `map()`, `ConditionalCheckFailedException` is caught in both `deleteRecipe` and `updateRecipe` to return 404, request bodies are validated for required fields before DynamoDB writes, and the auth guard waits for the session check to complete via a `sessionChecked$` Observable before making its routing decision.

**Watch for:** JSON parse failure in the Lambda returns a raw 500 with no structured error (confirmed), validation allows empty arrays for ingredients/instructions to pass (confirmed), `OnPush` components mutate state in subscribe callbacks without marking for check (confirmed ‚Äî works under Zone.js but fragile for zoneless migration).

**Verdict**: APPROVED

## High-level view

The Lambda's `validateRecipeInput` covers the four critical fields (title, servings, ingredients, instructions) with type checks, but the outer `JSON.parse` call has no try-catch ‚Äî a request with malformed JSON will throw a `SyntaxError` that bubbles to the top-level catch, returning a generic 500 instead of a 400 with a parse-related message. The validation also considers empty arrays valid for ingredients and instructions, so a recipe with zero ingredients will write to DynamoDB and render as a blank list in the UI.

The auth guard race condition from v1 is fully resolved ‚Äî `sessionChecked$` gates the routing decision until the async session check completes.

The `OnPush` + subscribe-mutation pattern remains unchanged across recipe components. Zone.js covers it today, but it's the first thing that breaks in a zoneless build. Acceptable for initial delivery per project scope.

1. **Malformed JSON body returns 500** ‚Äî `JSON.parse(event.body)` in `createRecipe` and `updateRecipe` throws `SyntaxError` on invalid JSON, which falls through to the top-level catch as a 500. Wrap the parse in a try-catch and return 400 with a "Malformed JSON" message.

## JSON parse error propagation

Both `createRecipe` and `updateRecipe` call `JSON.parse(event.body)` outside a try-catch:

```typescript
const parsed = JSON.parse(event.body);
const validationErrors = validateRecipeInput(parsed);
```

If the body is `"not json"` or truncated, `JSON.parse` throws `SyntaxError`. The top-level handler catch returns `response(500, { message: 'Internal server error' })`. From the client's perspective this looks like a server bug rather than a client error. The fix is a local try-catch around the parse that returns a 400:

```typescript
let parsed: unknown;
try {
  parsed = JSON.parse(event.body);
} catch {
  return response(400, { message: 'Request body is not valid JSON' });
}
```

Behind Cognito auth, only authenticated users can trigger it, but it still produces confusing 500s in server logs during client integration work.

## Validation boundary for empty collections

`validateRecipeInput` checks that `ingredients` and `instructions` are arrays:

```typescript
if (!Array.isArray(body['ingredients'])) {
  errors.push('ingredients is required and must be an array');
}
if (!Array.isArray(body['instructions'])) {
  errors.push('instructions is required and must be an array');
}
```

An empty array passes both checks. Whether this is a bug depends on product intent ‚Äî some applications allow saving recipe drafts with no ingredients. The frontend form starts with one ingredient row and requires a name (via `Validators.required`), so normal SPA usage won't produce empty arrays, but direct API callers could.
